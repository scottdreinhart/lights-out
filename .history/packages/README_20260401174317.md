# Shared Packages Workspace

This directory contains reusable cross-game packages that all game applications depend on and share.

Following CLEAN Architecture principles, these packages export pure, framework-agnostic logic and contracts that games compose into their applications.

## Current Packages

### Core Game Systems

- **`card-deck-core`** — Standard playing card system
  - Handles standard 52-card decks, jokers (53-54 cards), card values, numeric comparisons
  - Deck operations: shuffle, deal, burn, peek, reset
  - Multi-deck shoe support for casino-style games with penetration tracking
  - Unique card IDs for tracking across shoes
  - Pre-configured decks for Blackjack, Baccarat, Rummy, etc.
  - All games using cards should use this package

- **`domain-shared`** — Shared domain layer types and utilities
  - Common game types, constants, utilities
  - Board/grid helpers
  - Game rule validators

### UI & Display

- **`ui-utils`** — Shared UI helper functions
  - Component utilities, style helpers
  - Responsive design utilities

- **`display-contract`** — Contracts for rendering/display systems
  - Types and interfaces for display layer

- **`sprite-contract`** — Sprite system type contracts
  - Asset mapping and sprite definitions

- **`theme-contract`** — Theme/styling type contracts
  - Color tokens, typography, spacing

### React Integration

- **`app-hook-utils`** — Reusable React hooks for all games
  - Theme management, responsive state, sound effects
  - Game orchestration helpers
  - Input handling hooks

### Storage & Persistence

- **`storage-utils`** — localStorage & persistence helpers
  - Type-safe storage operations
  - Session management
  - Stats persistence

- **`stats-utils`** — Statistics tracking utilities
  - Score calculation
  - Win/loss tracking
  - Leaderboard helpers

### Error Handling & Logging

- **`crash-utils`** — Error handling & crash reporting
  - Centralized error logging
  - Recovery utilities
  - Diagnostic information

### Common Utilities

- **`common`** — General-purpose utilities
  - Type helpers, validation functions
  - Math utilities for game logic
  - UUID/ID generation

### Assets

- **`assets-shared`** — Shared sprite sheets and game assets
  - Common game piece graphics
  - UI iconography
  - Reusable tile sets

## Architecture Principles

### Dependency Flow

```
Each Game App (src/)
  ↓ imports from
Packages (shared logic, types, utilities)
  ↓
  ├─ Domain: Pure business logic (no React, framework-agnostic)
  ├─ App: React integration (hooks, context)
  ├─ UI: Shared components & contracts
  ├─ Storage: Persistence & retrieval
  └─ Common: General utilities
```

### No Circular Dependencies

- Packages import from other packages (downward only)
- Games import from packages (never upward)
- Packages never import from games

### Framework Agnostic Domain

All domain-layer packages (business logic, rules, types) contain **zero React code**:

- No hooks
- No JSX
- No framework dependencies
- Pure TypeScript/JavaScript

This ensures logic is:

- Independently testable
- Reusable across platforms (Web, Electron, Capacitor, Node)
- Portable to other frameworks

## Using Packages in Games

All packages are published under the `@games/` namespace and follow monorepo conventions.

### Import Examples

```typescript
// Card games
import { createDeck, shuffleDeck, dealCards } from '@games/card-deck-core'

// Shared UI utilities
import { useResponsiveState, useTheme } from '@games/app-hook-utils'

// Storage
import { storageService } from '@games/storage-utils'

// Domain logic
import { validateMove } from '@games/domain-shared'
```

## Adding New Packages

When adding a shared package:

1. **Name convention**: Kebab-case (e.g., `card-deck-core`)
2. **Directory structure**: `src/` with barrel `index.ts`
3. **Type definitions**: Include `src/types.ts` for public API
4. **Documentation**: Include `README.md` with examples
5. **No React required**: Domain logic should be framework-agnostic
6. **Barrel exports**: All public APIs re-exported from `src/index.ts`

Example new package:

```
packages/my-new-system/
├── package.json           # Name: @games/my-new-system
├── tsconfig.json
├── README.md
└── src/
    ├── types.ts          # Public types
    ├── core.ts           # Main logic
    ├── utils.ts          # Helpers (optional)
    └── index.ts          # Barrel export
```

## Monorepo Integration

All packages are part of the pnpm workspace (see `pnpm-workspace.yaml` at root).

```bash
# Install all packages
pnpm install

# Run scripts in a specific package
pnpm --filter @games/card-deck-core build

# Build all packages
pnpm build
```

Files are automatically available to games without publishing:

```json
// In game's package.json
{
  "dependencies": {
    "@games/card-deck-core": "workspace:*"
  }
}
```

## Migration & Consolidation

Over time, as games grow, common patterns should be extracted into shared packages:

- Similar AI engines → dedicated package
- Repeated UI patterns → consolidated atoms/molecules
- Game-specific types used across apps → domain package
- Platform-specific adapters → contract packages

This keeps games lightweight while maximizing reuse.
