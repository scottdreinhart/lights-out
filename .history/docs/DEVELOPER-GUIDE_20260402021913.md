## Getting Started

### Prerequisites

- **Node.js** v24+ (use [nvm](https://github.com/nvm-sh/nvm) and `.nvmrc`)
- **pnpm** v10+
- **Git** for version control
- **Bash / POSIX Shell** (WSL on Windows, native on macOS/Linux) — This is the mandatory default shell for all development

### Shell Governance (Mandatory)

**Bash (POSIX) is the mandatory default shell for all development work in this repository.**

- **Default**: Use Bash (WSL on Windows, native on macOS/Linux) for all commands: `pnpm install`, `pnpm dev`, `pnpm build`, linting, testing, etc.
- **Exception**: PowerShell is opt-in only and may only be used for `pnpm run electron:build:win` (Windows Electron packaging) when explicitly requested.
- **Never**: Assume PowerShell, generate PowerShell commands, or use Windows-native syntax by default, even on Windows machines.

For complete shell governance rules, see [AGENTS.md § 5](../../AGENTS.md#5-cross-platform-shell-governance-mandatory) (Cross-platform Shell Governance).

### Initial Setup

```bash
# Clone the repository
git clone <repo-url>
cd lights-out

# Install dependencies
pnpm install

# Verify setup
pnpm check          # Lint, format check, typecheck
pnpm build          # Verify build works
```

### Start Development

```bash
pnpm start          # Start dev server (http://localhost:5173)

# In another terminal
pnpm test           # Run unit tests (watch mode)
pnpm test:a11y      # Run accessibility tests
```

## Architecture Overview

### CLEAN Architecture (3 Layers)

```
UI Layer (src/ui/)
 ↓ imports
App Layer (src/app/)
 ↓ imports
Domain Layer (src/domain/)
```

#### Domain Layer (`src/domain/`)

**Pure, framework-agnostic logic** — zero React dependencies.

- `types.ts` — Type definitions (vocabulary)
- `constants.ts` — Configuration constants
- `rules.ts` — Game logic & rule enforcement
- `ai.ts` — AI/CPU decision-making
- `board.ts` — Board state management
- `themes.ts` — Theme & color data
- `responsive.ts` — Responsive breakpoints
- `index.ts` — Barrel export (public API)

**Properties**:
- Testable without React
- Reusable across platforms (Web, Electron, Mobile)
- Single responsibility per file
- No side effects (pure functions)

#### App Layer (`src/app/`)

**React integration layer** — state management, hooks, services.

- `useTheme.ts` — Theme management hook
- `useSoundEffects.ts` — Sound control hook
- `useResponsiveState.ts` — Responsive state hook
- `useGame.ts` — Game state orchestration
- `useStats.ts` — Statistics tracking
- `ThemeContext.tsx` — Theme provider
- `SoundContext.tsx` — Sound provider
- `storageService.ts` — localStorage wrapper
- `index.ts` — Barrel export (public API)

**Properties**:
- Bridges domain logic to React
- Never imports from `ui/`
- Provides dependency injection via Context
- Encapsulates side effects (API calls, storage)

#### UI Layer (`src/ui/`)

**Presentational components** — atoms, molecules, organisms.

##### Atoms (`atoms/`)
Smallest reusable primitives (Button, Input, Icon, Label, Badge, etc.).

- Pure presentational (no state logic)
- Accept all config via props
- Never import from `app/` or `domain/`
- Fully accessible & keyboard-navigable

##### Molecules (`molecules/`)
Composed atoms with light logic (FormGroup, MenuItem, HamburgerMenu, etc.).

- Compose 2-5 atoms together
- Simple local state (open/close, selected)
- Small custom hooks (< 30 lines)
- Connect to app hooks via props

##### Organisms (`organisms/`)
Full feature components (GameBoard, SettingsModal, StatusBar, etc.).

- Orchestrate molecules + atoms
- Integrate app hooks & context
- Handle user interactions
- 200-500 lines per component max

### Atomic Design Data Flow

```
Hooks (App) → Organism → Molecules → Atoms → Rendered UI
     ↓
   Props
   └─ (immutable, unidirectional)
```

## Code Style & Conventions

### Naming Conventions

| Pattern | Use Case | Example |
|---------|----------|---------|
| `use*.ts` | Custom React hooks | `useTheme.ts`, `useSoundEffects.ts` |
| `*Context.tsx` | React Context provider | `ThemeContext.tsx` |
| `*Service.ts` | Stateless utility | `storageService.ts` |
| `*.types.ts` | Type definitions | `types.ts` |
| `[A-Z]*.tsx` | React components | `GameBoard.tsx`, `Button.tsx` |
| `[a-z]*.ts` | Pure functions/logic | `board.ts`, `ai.ts`, `rules.ts` |
| `*.module.css` | Scoped component styles | `Button.module.css` |
| `*.css` | Global/theme styles | Theme styles in `src/themes/` |

### Import Patterns

```typescript
// ✅ GOOD: Use path aliases
import { useTheme } from '@/app'
import { COLOR_THEMES } from '@/domain'
import { GameBoard } from '@/ui'

// ❌ BAD: Don't use relative imports across layers
import { useTheme } from '../../app'
import { GameBoard } from '../UI/molecules'

// ✅ GOOD: Import from barrels only
import { Button } from '@/ui'

// ❌ BAD: Don't import internal files
import { Button } from '@/ui/atoms/Button/Button'
```

### TypeScript Guidelines

- **Strict mode enabled** — all code is strictly typed
- **No `any` types** — use explicit types or generics
- **Export types from barrels** — centralized type vocabulary
- **Type your props** — component props must be typed
- **Use `satisfies` operator** — for compile-time type checking without explicit casting

```typescript
// ✅ GOOD
interface ButtonProps {
  label: string
  onClick: () => void
  variant?: 'primary' | 'secondary'
}

export const Button = ({ label, onClick, variant = 'primary' }: ButtonProps) => {
  // ...
}

// ❌ BAD
export const Button = ({ label, onClick, variant }: any) => {
  // ...
}
```

### Code Organization

- **One component per file** (for organism/molecule)
- **Co-locate styles** — `Component.tsx` + `Component.module.css`
- **Max file size** — atoms < 100 lines, molecules < 200 lines, organisms < 500 lines
- **Extract hooks for reuse** — any logic in 2+ components → custom hook
- **No hardcoded values** — constants in `src/domain/constants.ts`, CSS variables for theming

## Development Workflow

### Local Development

```bash
# Terminal 1: Start dev server
pnpm start                  # http://localhost:5173

# Terminal 2: Run tests in watch mode
pnpm test                   # Auto-rerun on file changes

# Terminal 3: Run accessibility tests
pnpm test:a11y:debug       # Playwright inspector mode
```

### Making Changes

1. **Create feature branch** (from `develop`)
   ```bash
   git checkout -b feature/my-feature
   ```

2. **Make changes** following conventions above

3. **Validate locally**
   ```bash
   pnpm check              # Lint + format + typecheck
   pnpm build              # Verify production build
   pnpm test               # Run all tests
   ```

4. **Auto-fix issues**
   ```bash
   pnpm fix                # ESLint + Prettier on staged files
   ```

5. **Commit** (pre-commit hooks run automatically)
   ```bash
   git add .
   git commit -m "feat: add new feature"
   ```

### Pre-Commit Hooks

**Automatic on commit**:
- ESLint autofix on staged files
- Prettier formatting on staged files
- Husky prevents commits if linting fails

Cannot be bypassed; linting is mandatory.

## Testing & Quality

### Run All Checks

```bash
pnpm validate             # check + build (full pre-push gate)
pnpm check                # lint + format:check + typecheck
pnpm check:input          # Validate input control governance
```

### Individual Tools

```bash
# Linting
pnpm lint                 # ESLint check
pnpm lint:fix             # ESLint auto-fix
pnpm lint:metrics         # Show performance metrics

# Formatting
pnpm format               # Prettier auto-format
pnpm format:check         # Prettier validation only

# TypeScript
pnpm typecheck            # tsc --noEmit

# Testing
pnpm test                 # Jest (watch mode)
pnpm test:a11y            # Accessibility tests
```

### Bundle Analysis

```bash
pnpm analyze:report       # Generate bundle analysis
# View dist/bundle-report.html for interactive visualization
```

## Git & Commit Workflow

### Commits

- Use [Conventional Commits](https://www.conventionalcommits.org/):
  ```
  feat: add new feature
  fix: resolve issue
  docs: update documentation
  refactor: reorganize code without changing behavior
  test: add/update tests
  chore: dependency updates, CI/CD changes
  ```

- Keep commits atomic (one logical change per commit)
- Reference issues when applicable: `feat: #123 add feature`

### Pull Requests

1. **Branch naming**: `feature/`, `fix/`, `docs/`, `refactor/`
2. **PR description**: Explain "what" and "why"
3. **Link issues**: "Closes #123"
4. **All checks pass**: CI/CD validation required
5. **Code review**: At least one approval before merge
6. **Squash & merge** to `develop`

### Release Workflow

- **develop** → features & hotfixes
- **main** → production releases (tagged)
- Tag format: `v1.0.0` (semver)

## Deployment

### Web Deployment

```bash
# Build for production
pnpm build                # Output: dist/

# Deploy to hosting (e.g., Vercel, Netlify)
# CI/CD handles this automatically
```

### Electron Desktop

```bash
# Build for current platform
pnpm electron:build       # Platform-specific executable

# Build specifically
pnpm electron:build:win   # Windows .exe
pnpm electron:build:linux # Linux .AppImage
pnpm electron:build:mac   # macOS .dmg
```

### Mobile (Capacitor)

```bash
# Sync web build to native
pnpm cap:sync             # Updates native projects

# Deploy to device
pnpm cap:run:android      # Android app
pnpm cap:run:ios          # iOS app (macOS only)
```

## Troubleshooting

### Port 5173 Already in Use

```bash
pnpm dev                  # Auto-kills stale port
# or
kill-port 5173
```

### Node Modules Platform Issue (Windows/WSL)

Check `.node-platform.md`:
- If on WSL, run: `pnpm clean:node && pnpm install`
- If on Windows, run from PowerShell: same command

### Build Fails

```bash
pnpm clean:all            # Full reset
pnpm install
pnpm build                # Try again
```

### Tests Fail

```bash
pnpm test:a11y:debug      # Playwright inspector
# Review spec files in tests/ directory
```
